# Workflow — Homepage → Waitlist Conversion Audit

## Objective

Generate a prioritised list of hypotheses + concrete copy/structure diffs to improve the conversion rate from a homepage visit to a completed waitlist signup. Run on demand or on a recurring schedule. The output is a markdown file in `.tmp/` that the founder reviews and either implements or rejects.

This workflow does **not** make site edits directly. Its output is a recommendation document.

---

## When to run

- Every two weeks, or after any meaningful copy/structure change to the homepage or `/waitlist`.
- Immediately after a fresh batch of qualitative feedback lands (e.g. another `Haven - Launch feedback _ iterations` row).
- When PostHog metrics show a >20% week-over-week change in conversion (positive or negative — both are signal).

---

## Inputs (gather these first)

| Source | What | Where |
|---|---|---|
| Live homepage HTML | Current copy and structure | `index.html` |
| Live waitlist HTML | Current copy and structure | `waitlist/index.html` |
| Brand context | Positioning, voice, mission | `context.md`, `manifesto.md`, `four-layers-philosophy.md` |
| Brand guardrails | Rejection rules for off-brand recommendations | `workflows/brand-guardrails.md` |
| Latest qualitative feedback | What real signups said | `Haven - Launch feedback _ iterations - Sheet1.csv` (or any newer feedback file at repo root) |
| Quantitative funnel | PostHog metrics, last 30 days | `.tmp/metrics-{YYYY-MM-DD}.md` (generated by `tools/posthog_metrics.py`) |
| Bundled snapshot | Everything above in one document | `.tmp/snapshot-{YYYY-MM-DD}.md` (generated by `tools/site_snapshot.py`) |

If `.env` does not contain `POSTHOG_API_KEY` and `POSTHOG_PROJECT_ID`, run the audit on qualitative inputs only and clearly note "metrics unavailable" in the output.

---

## Steps

### 1. Snapshot

Run `python tools/site_snapshot.py`. This concatenates the live HTML + brand context + latest feedback CSV into `.tmp/snapshot-{date}.md`.

If the script fails (missing files, etc.), fix the script and update this workflow with the new constraint. Do not run the audit on partial inputs.

### 2. Pull metrics

Run `python tools/posthog_metrics.py --window 30d`. Outputs `.tmp/metrics-{date}.md` with:
- Homepage `$pageview` count
- CTA click-through rate (homepage → `/waitlist`)
- Waitlist form-start rate (anyone who landed on `/waitlist`)
- Signup completion rate (`waitlist_signed_up` ÷ form-starts)
- Drop-off between each stage
- Mobile vs desktop split for each stage

If the PostHog rate-limit kicks in (HTTP 429), the script backs off and retries. If retries fail, write `metrics: unavailable` to the output and continue with qualitative-only audit.

### 3. Skill-driven analysis

For each of the following, invoke the named skill against the snapshot. Do not re-implement what the skill does — let it run.

1. **`page-cro`** against the homepage section of the snapshot. Ask: where do parents lose interest, what's working, what's friction. Expect ~5–10 candidate recommendations.
2. **`signup-flow-cro`** against the `/waitlist` section. Ask: where in the form / confirmation flow do parents drop off, what's the friction. Expect ~3–5 candidates.
3. **`copy-editing`** on the highest-priority copy candidates from steps 1–2. Tone match, clarity, sentence length, em-dash usage, removal of trailing full stops on label-style headings.
4. **`marketing-psychology`** to sanity-check that the top candidates rest on a defensible mental model (loss aversion, social proof, etc.) — not just "make headline punchier".

### 4. Brand-guardrail cross-check

For each candidate from step 3, walk it through the five rules in `workflows/brand-guardrails.md`. Drop any candidate that violates a rule. Don't rewrite to fit — the rewrite tends to smuggle the violation back in.

Keep a "Rejected on guardrails" section in the output with each dropped candidate and the rule that killed it. This makes the guardrails visible and shows what the generic skills wanted to do that we vetoed.

### 5. Prioritise

Rank surviving candidates by **(potential lift × confidence) ÷ implementation cost**. Lift is qualitative — none of these are A/B tested yet. Be honest:
- High confidence: directly addresses something a real signup said in the feedback, OR addresses a clear funnel drop-off in the metrics.
- Medium confidence: aligned with established principle (e.g. above-the-fold value clarity) but no Haven-specific evidence yet.
- Low confidence: pattern-matched from generic CRO playbooks with no Haven evidence.

Output the top 5. More than 5 is noise — one human reviews this.

### 6. Output

Write to `.tmp/conversion-audit-{YYYY-MM-DD}.md` with this structure:

```markdown
# Conversion Audit — {date}

## TL;DR
{2–3 sentences: where the funnel is bleeding, what to try first}

## Funnel snapshot
| Stage | Count (30d) | Δ vs prior 30d | Conversion to next stage |
|---|---|---|---|
| Homepage views | ... | ... | ... |
| CTA clicks | ... | ... | ... |
| Form starts | ... | ... | ... |
| Signups | ... | ... | ... |

## Top 5 hypotheses

### 1. {Hypothesis}
- **Where:** {file:line or section name}
- **Current:** {quote of current copy/structure}
- **Proposed:** {concrete diff or new copy}
- **Why:** {evidence — qualitative or quantitative}
- **Rests on:** {mental model / principle}
- **Confidence:** high | medium | low
- **Effort:** S | M | L
- **How to test:** {A/B variant, change-and-watch, etc.}

(... repeat for hypotheses 2–5 ...)

## Rejected on guardrails
- {Rejected candidate} — violated Rule N: {one-line why}

## Notes
{Anything the agent learned this run that should improve future audits — write it back into this workflow as a numbered learning below.}
```

### 7. Stop

Don't implement anything. The founder reviews the output and decides what to ship.

## Rate-limit notes

- PostHog API: 240 req/min. The metrics tool batches into ~5 queries; well under the limit.
- If we ever start running this on every push, move it to a scheduled daily/weekly job to avoid hammering PostHog.

---

## Recurring schedule

This workflow is a one-shot when invoked. To run it on a schedule:
```
/schedule create "weekly conversion audit" --cron "0 9 * * MON" --prompt "Run workflows/conversion-audit.md and post the output as a PR comment"
```
Not enabled by default — opt in once the workflow has run cleanly twice on demand.
